  <name>Task 2: Create PerSourceRateLimiter with tenacity-based backoff + jitter</name>
  <files>src/soc_db/scraping/rate_limit.py</files>
  <action>
    Create `PerSourceRateLimiter` class in `src/soc_db/scraping/rate_limit.py`. This is a per-source rate limiter that uses tenacity for retry with exponential backoff + jitter, NOT a sliding-window counter like the existing `InMemoryRateLimiter` in `rate_limit.py`.

    Class signature:
    ```python
    class PerSourceRateLimiter:
        def __init__(
            self,
            source_id: str,                          # e.g. "wikipedia"
            requests_per_sec: float = 1.0,           # max sustained requests per second
            burst: int = 3,                          # max burst before throttling kicks in
            backoff_factor: float = 2.0,             # multiplier for exponential backoff
            max_retries: int = 5,                    # max retries before giving up
            min_wait: float = 0.5,                   # minimum wait between retries (seconds)
            max_wait: float = 60.0,                  # maximum wait between retries (seconds)
            jitter: bool = True,                     # add random jitter to wait times
        )
    ```

    Internal state:
    - `_last_request_time: float` — timestamp of last request
    - `_consecutive_failures: int` — count of consecutive failures
    - `_lock: threading.Lock` — thread safety (scrapers may run in threads in future)
    - `_source_id: str` — for logging

    Methods:

    a) `acquire(self) -> None`:
      - Synchronous method (since most scrapers are sync).
      - If time since `_last_request_time` is less than `1.0 / requests_per_sec`, sleep for the remainder.
      - Reset `_last_request_time` to `time.monotonic()` after sleeping.
      - Uses `time.sleep()` (not asyncio.sleep — scrapers are sync).

    b) `record_failure(self) -> None`:
      - Increment `_consecutive_failures`.

    c) `record_success(self) -> None`:
      - Reset `_consecutive_failures` to 0.

    d) `retry_decorator(self)`:
      - Returns a `tenacity.retry` decorator configured with:
        - `tenacity.retry_if_exception_type((ConnectionError, TimeoutError, IOError))`
        - `tenacity.stop_after_attempt(self.max_retries)`
        - `tenacity.wait_exponential(multiplier=self.backoff_factor, min=self.min_wait, max=self.max_wait)`
        - `tenacity.before_sleep` that logs the retry attempt with source_id
        - If `jitter=True`, wrap the wait with `tenacity.wait_random` equivalent: add `± 25%` jitter to calculated wait.
      - The decorator is applied to the fetch function, not the rate limiter itself.

    e) `__repr__(self) -> str`:
      - Returns `"PerSourceRateLimiter(source={source_id}, rate={requests_per_sec}/s)"`

    Use `tenacity` API:
    ```python
    from tenacity import (
        retry,
        retry_if_exception_type,
        stop_after_attempt,
        wait_exponential,
        before_sleep_log,
    )
    ```

    Logging: use `logging.getLogger(__name__)`. Log at WARNING level when retrying, INFO when acquiring.

    Import `random` for jitter: `wait = wait * (0.75 + random.random() * 0.5)` to get ±25% jitter.

    Do NOT import from any other `soc_db.scraping` module — rate_limit is a leaf module. Do NOT reference asyncio or async patterns — this is a synchronous rate limiter for synchronous scrapers.

    **Design rationale:** This rate limiter is simpler and more targeted than the existing sliding-window Redis-based limiter. That limiter protects API endpoints from client abuse (shared state across workers). This limiter protects target websites from scraper abuse (per-source, single-process, with backoff on failure).
