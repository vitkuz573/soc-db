    Step 3 — Create src/soc_db/wikidata.py with these components:

    3a. VENDOR_QIDS dict (mapping subset of VENDOR_FILES vendors to Wikidata QIDs):
    Source from scripts/scraper_wikidata_sparql.py MANUFACTURERS dict, plus additional
    vendors not covered there. For vendors without a known QID (e.g. Actions, ASPEED,
    Airoha, Altera, Amazon, Bitmain, Cavium, Nuvoton, Socionext, Sophgo, Synaptics,
    Tesla, Toshiba, SigmaStar, VIA WonderMedia, Cirrus Logic, APM), omit them from
    the dict — they will keep their hardcoded maps (fallback path).

    3b. SPARQL query builders:
    - `_build_process_query(vendor_qid: str) -> str` — queries P2175 (process node)
      on items with manufacturer (P178) = vendor_qid. Returns item label + process
      node value (nm). Use VALUES for P31 SoC classes from the legacy scraper.
    - `_build_gpu_query(vendor_qid: str) -> str` — queries P488 (GPU) on same
      manufacturer scope. Returns item label (model number) + GPU name label.
    - `_build_architecture_query(vendor_qid: str) -> str` — queries P10620 (instruction
      set architecture) or falls back to P1552 (has quality) with architecture
      qualifier. Returns vendor-level architecture string.

    Query design rules per per the CONTEXT.md Wikidata properties P2175/P488/P10620:
    - JOIN to labels via SERVICE wikibase:label with "en" language
    - LIMIT 500 per vendor, retry with broader query if 0 results
    - Each query returns bindings with both the item identifier (model) and the
      property value

    3c. `run_sparql(query: str) -> list[dict]`:
    - Uses SPARQLWrapper (not raw urllib) per CONTEXT.md requirement
    - Exponential backoff: 1s, 2s, 4s, 8s, max 16s (stop after 5 attempts)
    - Sets User-Agent to "SOC-DB/1.0 (+https://github.com/vitkuz573/soc-db)"
    - On persistent failure, log error and return [] (empty list triggers fallback)
    - Keyword arguments: `retries=5` with `base_delay=1.0`, `max_delay=16.0`

    3d. Query caching via `_cached_sparql(query: str, ttl: int = 86400) -> list[dict]`:
    - Reuse the same TTL-based cache pattern as `fetch()` in common.py:
      CACHE_DIR / md5(query).hexdigest as the cache key, check st_mtime,
      write new results as JSON-serialized binding list.
    - This avoids redundant network calls within the TTL window (default 24h).

    3e. `refresh_vendor_knowledge(dry_run: bool = False) -> dict[str, dict[str, Any]]`:
    - Iterates over vendors in VENDOR_QIDS that also exist in VENDOR_FILES
    - For each vendor: run process query, gpu query, architecture query
    - Assemble results into {vendor: {process_map: {...}, gpu_map: {...}, architecture: "..."}}
      matching VENDOR_KNOWLEDGE dict structure from _vendor_data.py
    - If dry_run=True: log results via logger.info per vendor (number of process
      mappings found, number of GPU mappings, architecture value) — do NOT write anywhere
    - If dry_run=False: return the assembled dict for the merge layer to consume
    - If ALL queries for a vendor fail (empty results), omit that vendor from
      returned dict (so fallback keeps hardcoded VENDOR_KNOWLEDGE)
    - Handle SPARQLWrapper.JSON format, parse `results["bindings"]`, extract
      value/label from each binding

    3f. Model-key normalization:
    - Process and GPU queries return (item_label, value) pairs
    - Map item_label to the model identifier used in VENDOR_KNOWLEDGE keys
      (e.g., "Snapdragon 8 Gen 2" with model "SM8550" → key "sm8550")
    - Use lowercase model for matching, same pattern as infer_gpu/infer_process

    2b. `merge_vendor_knowledge(wikidata_result: dict) -> dict`:
    - Takes the dict returned by `refresh_vendor_knowledge()` (Wikidata results)
    - Returns a merged copy of VENDOR_KNOWLEDGE where:
      1. Start with hardcoded VENDOR_KNOWLEDGE (full deep copy)
      2. For each vendor in wikidata_result:
         - If vendor NOT in hardcoded VENDOR_KNOWLEDGE: add the entire Wikidata entry
         - If vendor EXISTS: merge per-field:
           - architecture: replace with Wikidata value (Wikidata takes precedence for
             known items per CONTEXT.md)
           - process_map: for each key in Wikidata process_map, if key not in overrides
             for this vendor, set/update the value (Wikidata fills gaps, overrides win
             for specific models)
           - gpu_map: same merge logic as process_map
         Wikidata results fill in missing models but do NOT remove existing models
         that Wikidata didn't return (those stay from VENDOR_KNOWLEDGE)
      3. Apply overrides: for each vendor in overrides, for each field, for each key
         in the override, set the value unconditionally (manual correction wins over
         everything)
    - Returns the merged dict

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| soc-db process → Wikidata SPARQL endpoint | External HTTPS request to query.wikidata.org |
| GitHub Actions runner → Wikidata SPARQL endpoint | External HTTPS request from CI pipeline |
| vendor_overrides.json (disk) → merge layer | Local file read with untrusted (human-written) content |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-01 | Spoofing | SPARQL endpoint returns malicious data | medium | mitigate | merge_vendor_knowledge validates Wikidata values are str/int; overrides.json takes precedence for any known-bad data; dry-run before write gives human review opportunity |
| T-06-02 | Denial of Service | Wikidata rate-limit or block | low | accept | Exponential backoff handles transient failures; fallback to hardcoded VENDOR_KNOWLEDGE on persistent failure; cache avoids redundant queries |
| T-06-03 | Information Disclosure | SPARQL query exposes vendor QID mapping | low | accept | QIDs are public Wikidata identifiers; no sensitive data in queries |
| T-06-04 | Tampering | vendor_overrides.json corrupted | low | mitigate | load_overrides catches JSONDecodeError and returns {}; overrides file is git-tracked so corrupted state is recoverable |
| T-06-05 | Elevation of Privilege | GHA workflow creates PR with malicious content | low | accept | PR requires human review before merge per CONTEXT.md "never auto-publish"; GHA token is minimal (contents: write + pull-requests: write) |
