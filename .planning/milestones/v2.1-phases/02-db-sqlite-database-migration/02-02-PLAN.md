<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI user → CLI commands | Local user invokes CLI which reads from SQLite or JSON |
| API client → FastAPI | HTTP client sends requests that query SQLite via the data layer |
| SQLite → API response | Database rows get serialized to JSON for API responses |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-DB-06 | Information Disclosure | API search endpoint | low | accept | Search queries are user-provided; results contain only public SoC data. No access control required — data is intentionally public. |
| T-DB-07 | Elevation of Privilege | CLI cmd_enrich | low | mitigate | cmd_enrich always reads/writes JSON files, bypassing SQLite. This preserves the enrichment write path and prevents accidental SQLite writes from enrichment mutation. |
| T-DB-08 | Denial of Service | API FTS5 query | low | mitigate | FTS5 handles arbitrary input safely (parameterized). Long queries are limited by SQLite's max_query_length (default 1M). No regex injection possible through FTS5. |
| T-DB-09 | Spoofing | SOC_DB_USE_JSON env var | low | accept | Env var controls read backend only (no write path). Setting SOC_DB_USE_JSON=true simply falls back to JSON reads — no security boundary crossed. |
| T-DB-SC | Tampering | Stdlib only | high | mitigate | CLI and API use sqlite3 (stdlib) and aiosqlite (well-known PyPI). aiosqlite is [ASSUMED] — well-established package with 10M+ downloads. FTS5 is built into SQLite. No new risky dependencies. |

</threat_model>
