```
  └── _vendor_data.py ← VENDOR_KNOWLEDGE

```

## 4. Cross-Phase Integration Issues

### 4.1 BLOCKER — None Found

No completely broken cross-phase connections were identified. All exports are consumed, all API routes have callers, all CLI commands function with both backends.

### 4.2 WARNING — StatsResponse Type Mismatch

**Affected:** models.py ↔ queries.py  
**Files:** `src/soc_db/models.py:122`, `src/soc_db/db/queries.py:237,572`

`StatsResponse.fields_present` is typed as `dict[str, int]` but the SQLite query paths in `queries.py` return string values like `"123/1746"` instead of integers. The API `/v1/stats` endpoint has its own inline implementation that returns integers, so this doesn't cause a runtime error on the API path. However, the inconsistency could cause issues if:
- Code is refactored to delegate `/v1/stats` to `get_stats_async()`
- Someone feeds `queries.get_stats()` output into the Pydantic model

### 4.3 WARNING — Enrichment Architecture (By Design But Noted)

`enrich_one()` remains in `common.py` as the orchestrator that calls each per-domain `infer_*()` function. The ROADMAP describes "extract the 700-line `enrich_one()` into per-domain modules." The extraction has been done — the inference logic lives in per-domain files — but the orchestrator function remains in `common.py`. This is consistent with the documented shim/migration strategy but means `common.py` still has a 70-line `enrich_one()` function.

### 4.4 WARNING — No Phase Planning Directories

The GSD workflow `.planning/phases/` directory does not exist. No phase-specific PLAN.md or SUMMARY.md files were created. The ROADMAP shows phases 1, 2, 5 as "Planning" (not executed), but their code exists in the source tree. This means the code was implemented without going through the formal GSD phase workflow.

## 5. Tech Debt Identified

### P1 — WIKIDATA-02: Scheduled CI Runs Dry-Run Only

**Severity:** Medium  
**Affects:** Requirement WIKIDATA-02  
**File:** `.github/workflows/wikidata-refresh.yml` (line 33)

The scheduled weekly CI workflow runs `soc-db wikidata-refresh --dry-run` which only logs what would change — it never actually updates `VENDOR_KNOWLEDGE`. The non-dry-run path is only accessible via manual `workflow_dispatch` with explicit `dry_run=false`. This means:
- The "weekly refresh" is effectively a weekly validation, not an actual refresh
- No automatic PR is created with updated knowledge (the `create-pull-request` step runs but creates an empty PR since `--dry-run` doesn't modify files)

**Recommended fix:** Change the scheduled cron path to run without `--dry-run`, or add a second step that auto-merges if changes are detected.

### P2 — StatsResponse Type Inconsistency

**Severity:** Low  
**Affected:** `src/soc_db/models.py:122`, `src/soc_db/db/queries.py:237,572`, `src/soc_db/cli.py:228`

`fields_present` values are inconsistently typed: integers in JSON fallback paths, strings (`"123/1746"` format) in SQLite paths. The API endpoint avoids the issue with its own implementation, but this should be unified.

### P3 — No OpenTelemetry Overhead Benchmark

**Severity:** Low  
**Affected:** Success Criterion 5.4 (OBSERVE)

The OBSERVE phase success criteria require a before/after benchmark showing OTel overhead is under 5% with ~10% sampling. No such benchmark exists in the test suite.

### P4 — No FTS5 Search Quality Comparison

**Severity:** Low  
**Affected:** Success Criterion 2.3 (DB-02)

The DB phase success criteria require "equivalent or better search quality than v2.0 custom inverted index." No quantitative comparison exists. The FTS5 implementation is correct but unverified against the old index.

### P5 — Module-Level Mutability of `VENDOR_KNOWLEDGE`

**Severity:** Low  
**Affected:** `src/soc_db/enrich/_vendor_data.py`, `src/soc_db/cli.py:319-322`

`VENDOR_KNOWLEDGE` is mutated in-place at module level by both the conditional Wikidata import path (line 435-436) and the CLI `cmd_wikidata_refresh` (lines 319-322). Module-level mutable state can cause unexpected behavior in testing and concurrent access.

## 6. Circular Import Analysis

All cross-module imports were traced. The potential cycle is:

```
common.py → enrich/cpu.py → enrich/_vendor_data.py → (conditional) _vendor_data_wikidata.py → (lazy) wikidata.py → (lazy) enrich/_vendor_data.py
```

**Result: No circular imports.** All reverse edges (from wikidata and _vendor_data_wikidata back to _vendor_data) use **lazy function-level imports**, which execute after all modules are fully loaded. The conditional import in `_vendor_data.py` (line 431) is safe because `_vendor_data_wikidata.py` does **not** import from `_vendor_data` at module level.
